## ۳. محیط کاری Sandbox (Claude)

- Claude یک کامپیوتر لینوکسی جدا با دسترسی `bash_tool` داره.
- مخزن پروژه رو باید هر بار از گیت‌هاب clone کنه (چون هر sandbox جدید خالیه): `git clone https://github.com/AS228630/certnova-app.git`
- محدودیت شبکه‌ی `bash_tool`: فقط به چند دامنه‌ی خاص (`github.com`, `npmjs.com`, `pypi.org` و مشابه) دسترسی داره — نمی‌تونه مستقیم به Stripe، Supabase، یا خودِ certcoach.de وصل بشه.
- فایل‌های آپلودی کاربر (عکس‌ها، PDF) در `/mnt/user-data/uploads/` قرار می‌گیرن.
- فایل‌های خروجی (برای دانلود کاربر) باید در `/mnt/user-data/outputs/` کپی و با ابزار `present_files` نشون داده بشن.

## ۴. نحوه‌ی push کردن تغییرات به گیت‌هاب — ⚠️ نکته‌ی امنیتی حیاتی

Claude به‌تنهایی دسترسی push به گیت‌هاب نداره — نیاز به یک **GitHub Personal Access Token** داره که کاربر (شما) هر بار باید در چت وارد کنه:

```bash
git push https://<TOKEN>@github.com/AS228630/certnova-app.git main
```

### 🔴 قانون امنیتی که باید همیشه رعایت بشه:
**بعد از هر بار استفاده از یک توکن در چت، حتماً و بدون استثنا باید بلافاصله در `github.com/settings/tokens` آن توکن Revoke (باطل) بشه و برای push بعدی یک توکن تازه ساخته بشه.**

چرا این‌قدر مهمه: توکن همین‌که در متن چت تایپ می‌شه، دیگه «مخفی» نیست — تو تاریخچه‌ی چت باقی می‌مونه. در طول این پروژه **چندین بار** توکن‌ها بدون باطل شدن دوباره استفاده شدن (یه ریسک امنیتی واقعی که هنوز کامل رعایت نشده). هر چت جدید باید این هشدار رو به‌وضوح تکرار کنه.

### قدم‌های استاندارد قبل از هر push:
1. `npx tsc --noEmit` (بررسی خطای TypeScript)
2. `npx eslint <فایل‌های تغییریافته>` (بررسی سبک کد)
3. Build کامل با متغیرهای placeholder:
   ```bash
   cat > .env.local << 'EOF'
   NEXT_PUBLIC_SUPABASE_URL=https://placeholder.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=placeholder-anon-key
   EOF
   npm run build
   rm -f .env.local
   ```
4. فقط اگر هر سه مرحله بدون خطا بود: `git add -A && git commit -m "..." && git push ...`

### نکته‌ی فنی مهم درباره‌ی API Routes:
هرگز کلاینت‌های سرویس‌های خارجی (Stripe، Supabase Admin) رو در سطح بالای فایل نسازید — همیشه باید داخل تابع handler (lazy) ساخته بشن، وگرنه build کامل next خراب می‌شه چون متغیرهای محیطی در آن لحظه در دسترس نیستن.

## ۵. پایگاه‌داده (Supabase) — نکته‌ی حیاتی

Claude به هیچ‌وجه دسترسی مستقیم SQL به Supabase نداره — نمی‌تونه جدول بسازه یا migration رو خودش اجرا کنه. روند کار صحیح:

1. Claude یک فایل migration جدید در `supabase/migrations/0XX_نام.sql` می‌سازه (با شماره‌ی بعدی).
2. فایل رو در `/mnt/user-data/outputs/` کپی و با `present_files` نشون می‌ده.
3. **کاربر باید خودش** به `https://supabase.com/dashboard` بره → پروژه رو باز کنه → SQL Editor → New query → محتوا رو پیست کند → Run بزنه.
4. Claude همیشه باید مطمئن بشه کاربر واقعاً migration رو اجرا کرده، قبل از فرض کردن که یک جدول/ستون در دیتابیس وجود داره.

### جدول‌های موجود (تا این لحظه):
`user_topic_mastery`, `user_activity_log`, `support_messages`, `webinar_signups`, `newsletter_subscribers`, `subscriptions`, `user_cert_progress`, `user_lesson_completions`, `percert_lab_practice`, `profiles`, `exam_questions`, `ai_conversations`, `ai_messages`, `interview_prep`, `language_course_progress`, `community` (posts/likes/comments)`, `live_study_rooms`, `activity_read_state`, **`user_question_answers`** (جدید — پاسخ تک‌تک سؤالات تمرینی، برای حل‌شدن باگ قفل‌شدن بخش‌ها؛ **باید توسط کاربر اجرا بشه، فایل: `supabase/migrations/017_user_question_answers.sql`**)

## ۶. سیستم پرداخت Stripe — وضعیت فعلی

- کد سیستم پرداخت **کاملاً پیاده‌سازی شده** و امنه (checkout session، webhook، RLS، lazy client init — همه درست).
- پشتیبانی از Card + PayPal + Klarna از طریق یک ادغام Stripe واحد (نیازی به SDK جدا نیست).
- **کلید فعلی هنوز `pk_test_...` است — یعنی سایت هنوز روی Test mode است، نه Live.** برای پول واقعی، کاربر باید در Stripe Dashboard حساب رو Verify کنه و به Live mode سوییچ کنه.
- **PayPal تو صفحه‌ی پرداخت نمایش داده نمی‌شه** — چون تو Stripe Dashboard (نه در کد) فعال نشده. کاربر باید بره Settings → Payment methods و PayPal رو فعال کنه.
- کلید عمومی (Publishable Key) در `lib/stripeConfig.ts` امن است چون طراحی شده برای فرانت‌اند. کلیدهای مخفی (Secret Key، Webhook Secret، Supabase Service Role Key) فقط در Vercel Environment Variables هستن، هرگز در کد.

### دکمه‌ی کنسل اشتراک (قانونی، الزامی طبق § 312k BGB آلمان):
صفحه‌ی `/kuendigen` ساخته شده — همیشه در دسترس، بدون نیاز به لاگین، لینک شده تو فوتر. اشتراک رو واقعاً در Stripe لغو می‌کنه (نه فقط ظاهری).

| صفحه | وضعیت |
|---|---|
| Impressum | ✅ کامل، با اطلاعات واقعی |
| Datenschutz (حریم خصوصی) | ✅ کامل — همه‌ی ۵ مورد placeholder که قبلاً زرد و قابل‌مشاهده برای کاربر بودن، با اطلاعات واقعی و تأییدشده پر شدن (Vercel SCC، Supabase DPA، Stripe به‌عنوان پردازشگر پرداخت، Vercel Analytics بدون کوکی) |
| AGB (شرایط استفاده) | ✅ کامل نوشته شد (قبلاً فقط «به‌زودی» بود) — ۱۲ بخش، شامل قیمت، فسخ، مسئولیت |
| Widerrufsrecht (حق انصراف) | ✅ کامل — شامل بند § 356 Abs. 4 BGB (پایان زودهنگام حق انصراف برای خدمات دیجیتال) + فرم نمونه‌ی انصراف + چک‌باکس رضایت اجباری در چک‌اوت (با تاریخ/زمان در Stripe metadata ثبت می‌شه) |
| دکمه‌ی کنسل (§ 312k BGB) | ✅ صفحه‌ی `/kuendigen` ساخته شد |
| Barrierefreiheit / Sicherheit | ✅ صادقانه نوشته شدن (بدون ادعای دروغ درباره‌ی گواهی امنیتی یا سازگاری کامل با WCAG) |

**تنها توصیه‌ی باقی‌مونده:** قبل از فعال کردن نهایی پرداخت واقعی (Live mode)، یک‌بار این متون رو با یک وکیل آلمانی (یا سرویس آنلاین مثل eRecht24) چک کنید — این توصیه‌ی احتیاطی است، نه یک مشکل شناخته‌شده.

- ✅ **API چت KI Coach** قبلاً هیچ احراز هویتی نداشت (هر کسی بدون لاگین می‌تونست ازش استفاده و سهمیه‌ی پولی API رو هدر بده) — الان نیاز به توکن کاربر واقعی داره + محدودیت ۶۰ پیام در ساعت.

- ⚠️ **Content-Security-Policy (CSP) هنوز اضافه نشده** — عمداً به تعویق افتاده چون نیاز به فهرست دقیق تمام دامنه‌های خارجی (Vercel Analytics، Stripe، flagcdn.com، Supabase، EmailJS) داره تا چیزی رو خراب نکنه. یک قدم بعدی مشخص برای آینده.
- ✅ صفحه‌ی ۴۰۴ سفارشی (`app/not-found.tsx`) و error boundary (`app/global-error.tsx`) اضافه شدن — قبلاً صفحه‌ی پیش‌فرض بی‌هویت Next.js نمایش داده می‌شد.
- 🔴 **هیچ کلید مخفی در خودِ کد کامیت نشده** (چک شده، تمیزه) — اما توکن‌های گیت‌هاب چندین بار در متن چت افشا شدن. این نیازمند رعایت دقیق‌تر قانون بخش ۴ در آینده است.

7. بدون صفحه‌ی ۴۰۴/error سفارشی → اضافه شدن

### باگ‌های عملکردی
8. صفحه‌ی جزئیات دوره‌ی زبان (`/language-courses/[slug]`) اصلاً وجود نداشت → همه‌ی دکمه‌های «ادامه» ۴۰۴ می‌دادن → صفحه‌ی صادقانه‌ی «در حال ساخت» جایگزین شد
9. ۲ لینک مرده (`href="#"`) → یکی به داشبورد وصل شد، یکی (تزئینی تو شبیه‌سازی Azure) از پرش صفحه جلوگیری شد
10. **باگ بحرانی: پیشرفت بخش‌های تمرین (section unlock) هرگز در دیتابیس ذخیره نمی‌شد** — فقط در حافظه‌ی مرورگر بود و با بستن سایت پاک می‌شد → جدول `user_question_answers` + استور جدید ساخته شد، پاسخ‌ها حالا دائمی ذخیره می‌شن
11. متن سؤالات به‌شکل یک پاراگراف پیوسته نمایش داده می‌شد (بدون فاصله‌بندی pdf) → `whitespace-pre-line` اضافه شد
12. فونت گزینه‌های جواب خیلی کوچیک بود (۱۴px) → به ۱۶px افزایش یافت
13. سؤالات چندجوابی («select all that apply») فقط اجازه‌ی انتخاب یک گزینه رو می‌دادن → پشتیبانی کامل چک‌باکس/چندانتخابی اضافه شد

15. سؤالات تطبیقی (matching): ترتیب نمایش گزینه‌ها همیشه دقیقاً روبه‌روی جواب درستشون بود (قابل‌حدس بدون خوندن) → shuffle پایدار اضافه شد

### طراحی و UI (پیش‌تر در همین جلسه)

## ۱۲. اصل بنیادین پروژه — هیچ داده‌ی جعلی

این مهم‌ترین قانون کل پروژه است و باید در هر تصمیمی رعایت بشه:
- آمار پیشرفت کاربر همیشه باید از دیتابیس واقعی خونده بشه، هرگز عدد ساختگی.
- محتوای آموزشی هرگز نباید از سایت‌های شخص ثالث کپی بشه.
- اگه بخشی از یک ویژگی هنوز ساخته نشده، باید صادقانه «به‌زودی» یا حالت خالی نشون بده — نه محتوای جعلی.
- صفحات حقوقی فقط با اطلاعات واقعی که کاربر می‌ده پر بشن.
- قبل از نوشتن کد یا ساخت فایل، فایل‌های `SKILL.md` مرتبط در `/mnt/skills/public/` باید خونده بشن.

## نحوه‌ی استفاده از این فایل در چت بعدی

عطاالله جان، این فایل الان در ریشه‌ی مخزن گیت‌هاب (`PROJECT_GUIDE.md`) قرار داره. برای شروع هر چت جدید:

1. یا این فایل رو مستقیم از گیت‌هاب کپی کن و بفرست،
2. یا فقط بگو «فایل PROJECT_GUIDE.md رو از گیت‌هاب بخون» تا Claude با clone کردن ریپو خودش بخونتش.

**در پایان هر جلسه‌ی کاری، از Claude بخواه این فایل رو به‌روزرسانی و push کنه** — یه بخش جدید به «تاریخچه‌ی تغییرات» (بخش ۱۰) اضافه کنه، و بخش «قدم‌های باقی‌مانده» (بخش ۱۱) رو با وضعیت جدید هماهنگ کنه.

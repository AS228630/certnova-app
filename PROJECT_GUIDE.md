## ۴. نحوه‌ی push کردن تغییرات به گیت‌هاب — ⚠️ نکته‌ی امنیتی حیاتی

این محیط به‌تنهایی دسترسی push دائمی به گیت‌هاب نداره — نیاز به یک **GitHub Personal Access Token** داره که کاربر (شما) هر بار باید در چت وارد کنه:

```bash
git push https://<TOKEN>@github.com/AS228630/certnova-app.git main
```

### 🔴 قانون امنیتی که باید همیشه رعایت بشه:
**بعد از هر بار استفاده از یک توکن در چت، حتماً و بدون استثنا باید بلافاصله در `github.com/settings/tokens` آن توکن Revoke (باطل) بشه و برای push بعدی یک توکن تازه ساخته بشه.**

چرا این‌قدر مهمه: توکن همین‌که در متن چت تایپ می‌شه، دیگه «مخفی» نیست — تو تاریخچه‌ی چت باقی می‌مونه. در طول این پروژه **چندین بار** توکن‌ها بدون باطل شدن دوباره استفاده شدن (یه ریسک امنیتی واقعی که هنوز کامل رعایت نشده). هر چت جدید باید این هشدار رو به‌وضوح تکرار کنه.

### قدم‌های استاندارد قبل از هر push:
1. `npx tsc --noEmit` (بررسی خطای TypeScript)
2. `npx eslint <فایل‌های تغییریافته>` (بررسی سبک کد)
3. Build کامل با متغیرهای placeholder:
   ```bash
   cat > .env.local << 'EOF'
   NEXT_PUBLIC_SUPABASE_URL=https://placeholder.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=placeholder-anon-key
   EOF
   npm run build
   rm -f .env.local
   ```
4. فقط اگر هر سه مرحله بدون خطا بود: `git add -A && git commit -m "..." && git push ...`

### نکته‌ی فنی مهم درباره‌ی API Routes:
هرگز کلاینت‌های سرویس‌های خارجی (Stripe، Supabase Admin) رو در سطح بالای فایل نسازید — همیشه باید داخل تابع handler (lazy) ساخته بشن، وگرنه build کامل next خراب می‌شه چون متغیرهای محیطی در آن لحظه در دسترس نیستن.

## ۵. پایگاه‌داده (Supabase) — نکته‌ی حیاتی

این محیط به هیچ‌وجه دسترسی مستقیم SQL به Supabase نداره — نمی‌شه جدول ساخت یا migration رو خودکار اجرا کرد. روند کار صحیح:

1. یک فایل migration جدید در `supabase/migrations/0XX_نام.sql` ساخته می‌شه (با شماره‌ی بعدی).
2. فایل رو در `/mnt/user-data/outputs/` کپی و برای دانلود آماده می‌شه.
3. **کاربر باید خودش** به `https://supabase.com/dashboard` بره → پروژه رو باز کنه → SQL Editor → New query → محتوا رو پیست کند → Run بزنه.
4. همیشه باید مطمئن شد کاربر واقعاً migration رو اجرا کرده، قبل از فرض کردن که یک جدول/ستون در دیتابیس وجود داره.

### جدول‌های موجود (تا این لحظه):
`user_topic_mastery`, `user_activity_log`, `support_messages`, `webinar_signups`, `newsletter_subscribers`, `subscriptions`, `user_cert_progress`, `user_lesson_completions`, `percert_lab_practice`, `profiles`, `exam_questions`, `ai_conversations`, `ai_messages`, `interview_prep`, `language_course_progress`, `community` (posts/likes/comments)`, `live_study_rooms`, `activity_read_state`, **`user_question_answers`** (جدید — پاسخ تک‌تک سؤالات تمرینی، برای حل‌شدن باگ قفل‌شدن بخش‌ها؛ **باید توسط کاربر اجرا بشه، فایل: `supabase/migrations/017_user_question_answers.sql`**)

### مدیریت فضای دیتابیس (پلن رایگان Supabase = سقف ۵۰۰ مگابایت)

برای این‌که در آینده به مشکل فضا نخوریم، این اصل همیشه باید رعایت بشه:

- **داده‌ی ماندگار و باارزش کسب‌وکاری** (پروفایل کاربر، پیشرفت واقعی دوره/آزمون، اشتراک/پرداخت، پست‌های Community) → همیشه در جدول PostgreSQL واقعی ذخیره بشه (روش فعلی، درسته).
- **داده‌ی کاملاً موقتی و بدون نیاز به تاریخچه** (مثلاً وضعیت «کی الان آنلاینه»، نشانگر «در حال تایپ»، سیگنال‌های لحظه‌ای بین دو کاربر) → باید از **Supabase Realtime Presence/Broadcast** استفاده بشه، نه `insert()`. این‌ها هرگز روی دیسک ذخیره نمی‌شن، فقط در حافظه:

```js
// Presence — فقط در حافظه، هرگز روی دیسک
const channel = supabase.channel('live-room-123', {
  config: { presence: { key: 'user_session' } }
});
channel.track({ user: 'Ataullah', online_at: new Date() });

// Broadcast — پیام لحظه‌ای بدون ذخیره‌سازی
channel.send({ type: 'broadcast', event: 'chat-message', payload: {...} });
```

**نکته‌ی مهم:** چت اتاق‌های Live Coach عمداً در جدول واقعی ذخیره می‌شه (نه فقط Broadcast)، چون کاربر باید بتونه تاریخچه‌ی پیام‌ها رو دوباره ببینه. این یه تصمیم آگاهانه است، نه فراموشی این اصل.

## ۶. سیستم پرداخت Stripe — وضعیت فعلی

- کد سیستم پرداخت **کاملاً پیاده‌سازی شده** و امنه (checkout session، webhook، RLS، lazy client init — همه درست).
- پشتیبانی از Card + PayPal + Klarna از طریق یک ادغام Stripe واحد (نیازی به SDK جدا نیست).
- **کلید فعلی هنوز `pk_test_...` است — یعنی سایت هنوز روی Test mode است، نه Live.** برای پول واقعی، کاربر باید در Stripe Dashboard حساب رو Verify کنه و به Live mode سوییچ کنه.
- **PayPal تو صفحه‌ی پرداخت نمایش داده نمی‌شه** — چون تو Stripe Dashboard (نه در کد) فعال نشده. کاربر باید بره Settings → Payment methods و PayPal رو فعال کنه.
- کلید عمومی (Publishable Key) در `lib/stripeConfig.ts` امن است چون طراحی شده برای فرانت‌اند. کلیدهای مخفی (Secret Key، Webhook Secret، Supabase Service Role Key) فقط در Vercel Environment Variables هستن، هرگز در کد.

### دکمه‌ی کنسل اشتراک (قانونی، الزامی طبق § 312k BGB آلمان):
صفحه‌ی `/kuendigen` ساخته شده — همیشه در دسترس، بدون نیاز به لاگین، لینک شده تو فوتر. اشتراک رو واقعاً در Stripe لغو می‌کنه (نه فقط ظاهری).

## ۱۰. ⚠️ قانون دائمی — یک باگ که چند بار تکرار شد: هرگز `?? []` یا `?? {}` تازه از یه selector برنگردون

**این بخش رو قبل از هر کار روی هر Zustand store جدید یا موجود بخون.** این باگ **دو بار جداگانه** رخ داد (یک بار در `questionAnswersStore`، بار دوم در `sectionAttemptsStore` که بعداً اضافه شد) و هر بار باعث شد **کل صفحه‌ی تمرین برای همه‌ی گواهینامه‌ها** با صفحه‌ی سفید یا «Etwas ist schiefgelaufen» کرش کنه.

### علت دقیق
یه تابع getter داخل یه Zustand store که این شکلیه:
```ts
getSomething: (key) => get().someMap[key] ?? [],   // ❌ خطرناک
getSomething: (key) => get().someMap[key] ?? {},   // ❌ خطرناک
```
وقتی `key` هنوز داده‌ای نداره (مثلاً کاربر هنوز هیچ سؤالی رو جواب نداده)، هر بار که این تابع صدا زده می‌شه یه **آرایه/آبجکت کاملاً تازه با reference جدید** می‌سازه. React (از طریق `useSyncExternalStore` که Zustand زیرش استفاده می‌کنه) این reference جدید رو با قبلی مقایسه می‌کنه، می‌بینه فرق داره، دوباره رندر می‌کنه → که دوباره تابع رو صدا می‌زنه → که دوباره یه `[]`/`{}` تازه می‌سازه → **حلقه‌ی بی‌نهایت**. خطای دقیقش تو کنسول مرورگر اینه:
```
Warning: The result of getServerSnapshot should be cached to avoid an infinite loop
Maximum update depth exceeded
```
(در build فشرده‌شده، این به شکل «React error #185» ظاهر می‌شه.)

### فیکس (همیشه همین الگو رو دنبال کن)
یه ثابت پایدار **بیرون از تابع، در سطح ماژول** تعریف کن و به‌جای literal تازه ازش استفاده کن:
```ts
// در بالای فایل، خارج از create():
const EMPTY_LIST: SomeType[] = [];   // یا Record<string, X> برای آبجکت

// داخل store:
getSomething: (key) => get().someMap[key] ?? EMPTY_LIST,   // ✅ امن
```
این دقیقاً همون الگوییه که `certProgressStore.ts` از اول درست پیاده کرده بود (`EMPTY_DETAIL`) — منبع اون یکی درست بود، مشکل فقط تو ۲ فایل دیگه که بعداً نوشته شدن تکرار شد.

### قبل از merge کردن هر PR که یه Zustand store جدید اضافه یا ویرایش می‌کنه، این دستور رو اجرا کن:
```bash
grep -rn "^\s*get[A-Za-z]*:.*=>.*?? \[\]\|^\s*get[A-Za-z]*:.*=>.*?? {}" lib/store/*.ts
```
اگه خروجی داد، یعنی این باگ دوباره تکرار شده — قبل از commit کردن فیکسش کن.
